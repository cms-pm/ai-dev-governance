# Project Bootstrap Runbook

**Status:** normative for strict-baseline.

This runbook covers two paths: spinning up a new consumer project from
scratch ("new-project") and wiring `ai-dev-governance` into an existing
repo ("retrofit"). Both paths produce the same end state: a project that
passes `scripts/validate_bootstrap.sh`.

## Decision tree

```
Is this a brand-new repo?
│
├─ YES → use --new path (§New-project path)
│
└─ NO  → Does the repo already have ai-dev-governance as a submodule?
          │
          ├─ YES → use --retrofit path (§Retrofit path)
          │
          └─ NO  → Add the submodule first (§Add submodule manually),
                   then use --retrofit
```

---

## New-project path

Creates the full directory structure, adds submodules, writes wrappers,
initializes Astaire, and produces a bootstrap evidence bundle.

```bash
# From anywhere on your machine
scripts/bootstrap_project.sh \
  --new <target-dir> \
  --governance-url <your-governance-git-url> \
  [--with-board]
```

`--with-board` scaffolds `docs/governance/board/` and member directories.
Strict-baseline mode emits a warning when board scaffolding is skipped —
add it before beginning SCN work if the project uses board review.

After completion:

```bash
cd <target-dir>
scripts/validate_bootstrap.sh   # should pass clean
```

---

## Retrofit path

Brings an existing repo up to the bootstrap contract without touching
files outside it. The default is **dry-run** — it reports what would
change without writing anything.

```bash
# Dry-run (safe to run at any time — shows diff report only)
.governance/ai-dev-governance/scripts/bootstrap_project.sh --retrofit

# Apply changes
.governance/ai-dev-governance/scripts/bootstrap_project.sh --retrofit --force

# With board scaffolding
.governance/ai-dev-governance/scripts/bootstrap_project.sh \
  --retrofit --force --with-board
```

Files outside the bootstrap contract are never touched. `AGENTS.md` /
`CLAUDE.md` updates are scoped to the managed marker block:

```
<!-- ai-dev-governance:bootstrap:start -->
...managed content...
<!-- ai-dev-governance:bootstrap:end -->
```

Content outside the markers is preserved on every run.

## Add submodule manually

If you cannot run `bootstrap_project.sh --new` (for example, when
retrofitting a repo that has its own CI constraints on submodule adds):

```bash
# 1. Add governance submodule
git submodule add <your-governance-git-url> .governance/ai-dev-governance
git submodule update --init --recursive

# 2. Then retrofit
.governance/ai-dev-governance/scripts/bootstrap_project.sh --retrofit --force
```

## Verify a bootstrapped project

```bash
scripts/validate_bootstrap.sh
# or equivalently:
.governance/ai-dev-governance/scripts/bootstrap_project.sh --verify
```

Checks performed:

| Check | What it verifies |
|---|---|
| `.astaire/astaire` present & executable | Wrapper exists and is runnable |
| `.astaire/memory_palace.db` gitignored | DB not accidentally committed |
| `governance.yaml` with required keys | Manifest is structurally valid |
| `AGENTS.md` / `CLAUDE.md` bootstrap block | Astaire CLI surface loaded at session start |
| Required directory structure | `docs/planning/`, `docs/releases/`, `docs/governance/` |
| Governance submodule initialized | Submodule checked out, not empty |
| Astaire DB initialized (soft) | `.astaire/astaire startup` has been run |
| Tentacle pin (soft) | Astaire SHA matches `COMPATIBILITY_MATRIX.md` |

---

## Pin-aware behavior

`bootstrap_project.sh` reads tentacle pins from
`COMPATIBILITY_MATRIX.md` at runtime. If the checked-out Astaire
submodule SHA does not match the matrix, the script refuses to run
(in `--new` mode) or emits a pin-mismatch failure (in `--force`
retrofit mode). Update the submodule pin before bootstrapping:

```bash
cd .governance/ai-dev-governance/astaire
git fetch --tags
git checkout <expected-sha>
cd -
git add .governance/ai-dev-governance/astaire
git commit -m "Pin astaire to <expected-sha>"
```

## What the bootstrap contract covers

| Item | Notes |
|---|---|
| `.astaire/astaire` | Wrapper; delegates to governance submodule |
| `.astaire/memory_palace.db` | Gitignored; regenerated by `startup` |
| `governance.yaml` | Never overwritten in retrofit; created if missing |
| `AGENTS.md` (or `CLAUDE.md`) | Bootstrap block updated in place via markers |
| `docs/planning/pool_questions/` | Created if missing |
| `docs/releases/` | Created if missing |
| `docs/governance/amendments/` | Created if missing |
| `docs/governance/board/**` | Created only with `--with-board` |
| `docs/releases/bootstrap/<date>-bundle.md` | Evidence written on every `--new` or `--retrofit --force` |

---

## Bootstrap evidence bundle

Every `--new` or `--retrofit --force` run writes a timestamped evidence
bundle to `docs/releases/bootstrap/<ISO-date>-bundle.md`. The bundle
enumerates every wire-up check with pass/fail and timestamps. It is the
release evidence artifact for SCN-D-04.
